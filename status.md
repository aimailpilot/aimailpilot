## Golden Rules (NEVER violate)

- **NEVER** import `server/db.ts` ‚Äî crashes production (`drizzle-orm` not installed, caused 1 day downtime)
- **NEVER** drop or recreate tables ‚Äî use `ALTER TABLE ADD COLUMN` with try/catch only
- **NEVER** `setContacts([])` on API error ‚Äî keep stale data visible (caused "No contacts" display TWICE)
- **NEVER** bypass `storage` methods for working GET endpoints ‚Äî raw SQL only as enhancement after storage fetch
- **NEVER** modify `sendViaGmailAPI` return value ‚Äî must return `{ success, messageId, threadId }` for threading
- **NEVER** remove `gmailThreadId` save in `sendBatched()` or lookup in `executeFollowup()`
- **NEVER** remove 401 retry + force-refresh in `sendEmail()` Gmail path in `followup-engine.ts`
- All camelCase columns in PostgreSQL raw SQL MUST be double-quoted: `"organizationId"`, `"contactId"`, `"createdAt"`, etc.
- Raw SQL only via `storage.rawGet/rawAll/rawRun` ‚Äî auto-converts `?` to `$1,$2` for PG
- **PostgreSQL returns `COUNT(*)` and `SUM()` as bigint strings** ‚Äî always wrap with `parseInt()` / `Number()` or JS will string-concatenate instead of add
- `email_accounts` uses `"isActive" = 1` column (NOT `status = 'active'` ‚Äî that column does not exist)
- `fromEmail` in `unified_inbox` stores `"Name <email>"` format ‚Äî extract email before comparing
- **Inbox Reply tab** filter must be `(status = 'replied' OR "repliedAt" IS NOT NULL)` in all 3 locations in `pg-storage.ts` ‚Äî `status = 'replied'` alone shows empty Reply tab
- **Forward endpoint** must set `forwardedFrom: msg.fromEmail` ‚Äî required so original sender email shows in UI
- **Bounce classifier** (`reply-classifier.ts`) non-system senders require 3+ indicators to classify as bounce ‚Äî do NOT lower threshold (legitimate replies were being misclassified)
- **Campaign enrichment** in `GET /api/campaigns` ‚Äî `Promise.all` async map with try/catch ‚Äî do NOT remove (powers "Sent By" and "List" columns)
- **Reply classification rules** in `reply-classifier.ts` ‚Äî 30+ strengthened patterns for OOO, auto_reply, positive, negative, bounce. Do NOT weaken patterns or lower thresholds. Non-system senders require 3+ indicators for bounce classification (legitimate replies were being misclassified). Do NOT change `isHumanReply()` logic which identifies actionable replies.
- **Boot reclassification** in `server/index.ts` ‚Äî targets `("replyType" IS NULL OR "replyType" = '' OR "replyType" = 'general')` at 30s startup delay. Do NOT change delay timing or message limits. Do NOT remove AI reclassification fallback for borderline cases.
- **Inbox reply filter** ‚Äî "Emails Need Reply" and "Not Replied" filters must exclude non-human replies: `replyType IN ('positive', 'negative', 'general')` only. Do NOT revert to include-based filters. OOO/auto_reply/bounce must always be excluded.
- **Scorecard "Not Replied" cutoff** in `routes.ts` ‚Äî must enforce 3-day lookback via `notRepliedCutoff` to allow reply time. Do NOT remove cutoff or revert to all-time counts. Gives accurate signal for aged emails.
- **Email rating badge condition** in `contacts-manager.tsx` ‚Äî use `contact.emailRatingUpdatedAt` (not `emailRating > 0`) to detect rated contacts. A 0/F rating is valid and must show a badge.
- **`messages.recipientEmail` backfill** in `pg-storage.ts initializeSchema()` ‚Äî runs after schema COMMIT using `queryOne`/`execute` (not `client`). Backfills from `contacts` table where `contactId` still resolves. Do NOT remove ‚Äî required for rating fallback on re-imported contacts.
- **Batch rating is fire-and-forget** ‚Äî `POST /api/contacts/batch-rating` responds immediately with `background: true`, then runs `batchRecalculateRatings()` via `setImmediate()`. Do NOT make it synchronous (causes HTTP timeout on large orgs).
- **autoPaused flag** ‚Äî `campaigns.autoPaused` MUST be set to `true` on ALL system-initiated pauses (window closed, daily limit, sendBatched crash) and `false` on ALL user-initiated pauses (UI pause/stop). Boot recovery SQL filter depends on this being correct. Do NOT change `autoPaused` logic in `campaign-engine.ts` or `routes.ts` pause/stop endpoints.
- **resumeActiveCampaigns two-pass** ‚Äî Pass 1 (`status='active'`) and Pass 2 (`status='paused' AND autoPaused=true AND autopilot.enabled=true`) are both required. Do NOT collapse into one query or remove either pass. Pass 2 recovers stranded campaigns after server restart.
- **resumeActiveCampaigns boot stagger** ‚Äî 2s `await` delay after each `startCampaign()` call in both passes. Do NOT remove. Prevents simultaneous contact loading from causing heap OOM on boot with many campaigns.
- **contactIds zero-contact abort** ‚Äî In `startCampaign()`, if `getContactsByIds()` returns 0 contacts, return error immediately. Do NOT revert to `getContacts(org, 10000, 0)` fallback ‚Äî loads all org contacts (188k+) and causes OOM.
- **Follow-up tracking injection** ‚Äî `followupTrackingId` must be generated BEFORE `sendEmail()` call in `executeFollowup()`. `addTrackingPixel()` + `addClickTracking()` must be applied to `personalizedContent` before sending. The same ID must be stored in the message record and tracking event. Do NOT move generation after send or skip injection.
- **Follow-up base URL** ‚Äî `FollowupEngine.setPublicBaseUrl()` MUST be called in `routes.ts` alongside every `campaignEngine.setPublicBaseUrl()` call (7 sites total). Without this, `getBaseUrl()` falls back to `http://localhost:8080` on Azure (no `BASE_URL` env var), baking broken localhost URLs into tracking pixels of all Step 2+ emails. Confirmed root cause of 0 opens/clicks on follow-up steps (fixed 2026-04-19).
- **Gmail/Outlook warmup filter** ‚Äî `ownEmailsSet` in `gmail-reply-tracker.ts` and `outlook-reply-tracker.ts` must be built from `email_accounts` (direct) AND `warmup_accounts` JOIN `email_accounts` (via `emailAccountId`). `warmup_accounts` has NO `email` column ‚Äî must JOIN `email_accounts ea ON ea.id = wa."emailAccountId"`. Using `SELECT email FROM warmup_accounts` returns nothing and breaks the skip (root cause of 32k warmup inbox flood, fixed 2026-04-18).
- **Gmail 404 handling** ‚Äî `gmailFetch()` returns `null` on 404. All callers that destructure the result must check `if (!msg) continue` first. Do NOT add 404 to the retry path.
- **Replied tab filter** ‚Äî overridden 2026-04-18 to `"replyType" IN ('positive','negative','general')`. Do NOT revert to `(status='replied' OR "repliedAt" IS NOT NULL)` ‚Äî showed only 1 message vs thousands of real replies.
- **Warmup inbox exclusion** ‚Äî `getInboxMessagesEnhanced`, `getInboxMessageCountEnhanced`, `getInboxStats` in `pg-storage.ts` exclude warmup by sender only (`fromEmail NOT IN orgEmails`). `orgEmails` must union both `email_accounts` AND `warmup_accounts`. Do NOT revert to email_accounts-only or to both-sides (fromEmail AND toEmail) check.
- **getUnrepliedCampaignMessages 45-day cutoff** ‚Äî `pg-storage.ts` ‚Äî `AND m."sentAt" >= $2` with 45-day window is required. Removing reverts to full table scan (8s+ PG-SLOW). Do NOT remove or extend to all-time.
- **getInboxStats accountIds scoping** ‚Äî `pg-storage.ts getInboxStats(orgId, accountIds?)` ‚Äî when `accountIds` is provided, ALL count queries must include `AND "emailAccountId" IN (...)`. Routes MUST pass member account IDs for non-admin roles. Do NOT call with no accountIds for members ‚Äî returns org-wide counts that leak other members' data.
- **Reply Quality auto-scorer** ‚Äî `server/index.ts` 15-min sweep (`batchScoreOrgReplies`) runs 90s after boot. Do NOT remove ‚Äî without it, `replyQualityScore` never populates and dashboard hot/warm breakdowns show zeros. Rule-based fallback ensures Azure outage doesn't block scoring.
- **Reply quality SQL uses FILTER syntax** ‚Äî `COUNT(*) FILTER (WHERE "replyQualityLabel" = 'Hot')` requires PostgreSQL 9.4+. Do NOT rewrite to `CASE WHEN` subqueries unless targeting SQLite compat.
- **Warmup DB purge safety** ‚Äî `warmup-inbox-purge.ts` DELETE must include `replyType NOT IN ('positive','negative','general')` guard. Do NOT remove ‚Äî prevents real human replies from being purged even if sender/recipient are both org accounts.
- **Warmup inbox cleanup** ‚Äî `warmup-inbox-cleanup.ts` only removes INBOX label from messages already carrying `AImailPilot-Warmup` label. Never touches real client mail. Do NOT add deletions to this service.
